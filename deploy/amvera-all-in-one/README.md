# Vibe Proxy Nexus — единый контейнер для Amvera

Этот пакет собирает **весь проект в одном Docker-образе** для Amvera Cloud:

- **Xray-core** — VPN-сервер (VLESS-XTLS-Reality) на порту `443`;
- **Node/Express** — API **и** собранный веб-интерфейс на одном порту (`8080`).

Оба процесса запускаются параллельно через `supervisord`. Поскольку бэкенд и
Xray находятся в одном контейнере, при выдаче/отзыве ключа бэкенд напрямую правит
конфиг Xray на диске и перезапускает процесс — отдельный management API не нужен.

Снаружи контейнера остаётся только:

- **PostgreSQL** — управляемая база Amvera (передаётся через `DATABASE_URL`).

Авторизация (email + пароль) реализована прямо в бэкенде и хранит сессии в
той же PostgreSQL — никаких внешних сервисов для входа не нужно.

Файлы `Dockerfile` и `amvera.yml` лежат в **корне репозитория** (Amvera по
умолчанию ищет их там); контекст сборки — корень репозитория.

## Как это работает

```
                 ┌───────────────────────── контейнер ─────────────────────────┐
  VPN-клиент ───▶│  Xray-core  :443  (Reality)                                  │
                 │                                                              │
  Браузер ──────▶│  Node/Express  :8080  ──▶  /api/*        (API)              │
                 │                        └─▶  всё остальное (React SPA)        │
                 │        ▲                                                     │
                 │        └── правит /etc/xray/config.json + supervisorctl      │
                 └──────────────────────────────────────────────────────────────┘
                          │
                          ▼
                  PostgreSQL (Amvera)
```

## Переменные окружения

### Секреты (в панели Amvera)

| Переменная            | Назначение                                                        |
| --------------------- | ------------------------------------------------------------------ |
| `DATABASE_URL`        | строка подключения к PostgreSQL от Amvera                          |
| `SESSION_SECRET`      | секрет для подписи cookie сессии, например `openssl rand -hex 32`  |
| `REALITY_PRIVATE_KEY` | приватный ключ Reality (см. ниже)                                  |
| `REALITY_PUBLIC_KEY`  | публичный ключ Reality — вносится в узел в админке                 |
| `REALITY_SHORT_ID`    | short id Reality                                                    |

### Обычные значения

| Переменная    | По умолчанию          | Назначение                              |
| ------------- | --------------------- | --------------------------------------- |
| `REALITY_SNI` | `www.microsoft.com`   | домен-маскировка для Reality            |
| `PORT`        | `8080`                | порт веб-интерфейса + API               |

## Подготовка перед первым деплоем

1. **Ключи Reality.** Сгенерируйте пару (можно локально любым Xray):

   ```sh
   docker run --rm ghcr.io/xtls/xray-core x25519
   ```

   `Private key` → `REALITY_PRIVATE_KEY`, `Public key` → `REALITY_PUBLIC_KEY`.
   `REALITY_SHORT_ID` — произвольная hex-строка, например `openssl rand -hex 8`.

2. **База данных.** Создайте в Amvera управляемый PostgreSQL и возьмите его
   `DATABASE_URL`. Один раз накатите схему (из этого репозитория, указав в
   окружении нужный `DATABASE_URL`):

   ```sh
   DATABASE_URL="<amvera-postgres-url>" pnpm --filter @workspace/db run push
   ```

3. **Секрет сессии.** Сгенерируйте случайную строку для `SESSION_SECRET`:

   ```sh
   openssl rand -hex 32
   ```

## Деплой

1. Залейте репозиторий в Amvera (git push или через интерфейс).
2. Проставьте секреты в настройках приложения.
3. Amvera соберёт образ по корневому `Dockerfile` и запустит контейнер.

## После деплоя

1. Откройте веб-интерфейс по домену приложения Amvera.
2. Зайдите как администратор, в разделе **Узлы (VPN nodes)** создайте узел:
   - `host` — домен вашего приложения Amvera (куда подключаются VPN-клиенты);
   - `sni` — то же значение, что `REALITY_SNI`;
   - `publicKey` — `REALITY_PUBLIC_KEY`;
   - `shortId` — `REALITY_SHORT_ID`.
3. Теперь выданные из панели ключи будут автоматически добавляться в живой Xray.

## Порты и один IP

По умолчанию VPN слушает `443`, а веб — `8080`.

**Важно про `amvera.yml`**: в схеме Amvera нет поля `run.ports` (список) —
только одиночный `run.containerPort` (число). Через него наружу на публичный
домен (`https://...amvera.tech`) отдаётся ровно один порт — веб/API (`8080`).
Порт `443` (сырой TCP для VLESS-XTLS-Reality) этим механизмом не пробрасывается
вообще; чтобы VPN был доступен из интернета, нужен отдельный аддон
**«Dedicated IPv4»** / ручной маппинг порта в разделе «Домены» панели Amvera
(не через YAML).

Раздел «Домены» → форма «URL путь» + «Порт»: предупреждение платформы «не
заполняйте, если используете один порт» относится именно к `containerPort` —
если он один (сейчас так и есть), это поле трогать не нужно, иначе можно
поймать «Ошибку конфигурации».

Если нужно, чтобы и панель, и VPN работали на одном порту `443` (один IP, один
порт), это делается через механизм `fallbacks` Xray — расширенная настройка,
которая требует собственного TLS-сертификата для домена панели и здесь намеренно
не включена, чтобы не усложнять базовый сценарий.

## Локальная сборка (проверка)

```sh
docker build -t vibe-proxy-nexus .

docker run --rm -p 8080:8080 -p 443:443 \
  -e DATABASE_URL=... \
  -e SESSION_SECRET=... \
  -e REALITY_PRIVATE_KEY=... \
  -e REALITY_SHORT_ID=... \
  vibe-proxy-nexus
```

## Отличие от `deploy/amvera-vpn-node`

- **`deploy/amvera-all-in-one`** (этот пакет) — всё в одном контейнере.
  Рекомендуется для одного узла.
- **`deploy/amvera-vpn-node`** — только VPN-узел (Xray + management API),
  для будущего масштабирования на несколько стран, когда узлы выносятся на
  отдельные машины, а панель остаётся одна.
