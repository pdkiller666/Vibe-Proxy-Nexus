# Vibe Proxy Nexus — единый контейнер для Amvera

Этот пакет собирает **весь проект в одном Docker-образе** для Amvera Cloud:

- **Xray-core** — VPN-сервер (VLESS+WS), слушает только `127.0.0.1:10000` внутри
  контейнера, наружу напрямую не торчит;
- **Node/Express** — API, собранный веб-интерфейс **и** прокси
  WebSocket-апгрейда до Xray, всё на одном публичном порту (`8080`).

Оба процесса запускаются параллельно через `supervisord`. Поскольку бэкенд и
Xray находятся в одном контейнере, при выдаче/отзыве ключа бэкенд напрямую правит
конфиг Xray на диске и перезапускает процесс — отдельный management API не нужен.

Снаружи контейнера остаётся только:

- **PostgreSQL** — управляемая база Amvera (передаётся через `DATABASE_URL`).

Авторизация (email + пароль) реализована прямо в бэкенде и хранит сессии в
той же PostgreSQL — никаких внешних сервисов для входа не нужно.

Продакшен-домены: `https://vibeproxynexus-pdkiller666.waw0.amvera.tech/` и `https://vpnexus.pro/`.

Файлы `Dockerfile` и `amvera.yml` лежат в **корне репозитория** (Amvera по
умолчанию ищет их там); контекст сборки — корень репозитория.

## Почему VLESS едет поверх WebSocket, а не сырым TCP/Reality

Amvera-е edge (Traefik) на публичном HTTP(S)-порту **всегда** терминирует TLS
сам и ожидает за собой обычный HTTP. Это значит:

- **Reality несовместим в принципе** — протокол требует, чтобы именно Xray
  сам держал TLS-хендшейк, а тут TLS уже снят чужим прокси.
- **Сырой VLESS через TCP-контроллер Amvera** (домены `MONGO`/`POSTGRES`/
  `REDIS`) технически даёт TCP-passthrough, но от этой схемы отказались —
  требует отдельного TCP-домена, нестандартного порта.

Рабочая схема — **VLESS поверх WebSocket на обычном HTTPS-домене**:

1. Xray слушает `127.0.0.1:10000` с `network: ws`, `security: none`, `path: /vpnws`.
2. Node-сервер обрабатывает HTTP `upgrade`-события: если путь `/vpnws` —
   открывает TCP до `127.0.0.1:10000` и прозрачно проксирует байты.
3. Клиент подключается: `security=tls&type=ws&sni=<домен>&path=/vpnws` на порт 443.

```
                 ┌───────────────────────── контейнер ─────────────────────────┐
  Браузер ──────▶│  Node/Express  :8080  ──▶  /api/*             (API)         │
  VPN-клиент ───▶│  (тот же порт)         ├─▶  /vpnws (WS upgrade) ─┐          │
                 │                        └─▶  всё остальное (React SPA)      │
                 │                                                  ▼          │
                 │                                    Xray-core :10000 (loopback)
                 │        ▲                                                    │
                 │        └── правит /etc/xray/config.json + supervisorctl     │
                 └──────────────────────────────────────────────────────────────┘
                          │
                          ▼
                  PostgreSQL (Amvera)
```

История попыток и обоснование — `.agents/memory/amvera-raw-tcp-port.md`.

## Переменные окружения

### Секреты (в панели Amvera)

| Переменная       | Назначение                                                        |
| ---------------- | ------------------------------------------------------------------ |
| `DATABASE_URL`   | строка подключения к PostgreSQL от Amvera                          |
| `SESSION_SECRET` | секрет для подписи cookie сессии **и** для HMAC-подписи ссылок-подписок (`openssl rand -hex 32`) |
| `ADMIN_EMAIL`    | email первого администратора — создаётся автоматически при первом старте |
| `ADMIN_PASSWORD` | пароль для этого администратора |

### Обычные значения

| Переменная | По умолчанию | Назначение |
| ---------- | ------------ | ---------- |
| `PORT`     | `8080`       | единственный публичный порт: веб + API + VPN (WebSocket-прокси до Xray) |

Ключи Reality/Xray (`publicKey`, `shortId`, `sni`, `host`, `port`) хранятся в
таблице `vpn_nodes` и вводятся через админ-панель — не нужны как env-переменные.

## Подготовка перед первым деплоем

1. **База данных.** Создайте в Amvera управляемый PostgreSQL, возьмите `DATABASE_URL`.
   Схему накатывать вручную не обязательно — `entrypoint.sh` делает `drizzle-kit push`
   при каждом старте (идемпотентно).

2. **Секрет сессии:**
   ```sh
   openssl rand -hex 32
   ```

3. **Учётные данные первого администратора** — задайте `ADMIN_EMAIL` и
   `ADMIN_PASSWORD` в панели Amvera → «Переменные» (не в `amvera.yml`).

## Деплой

Из Replit — командой `./deploy.sh "Сообщение на русском о том, что изменилось"`.
Amvera сама подхватывает пуш в GitHub и пересобирает образ.

## После деплоя

1. Войдите под `ADMIN_EMAIL`/`ADMIN_PASSWORD`.
2. В разделе **Реквизиты** заполните СБП-реквизиты для приёма оплаты.
3. В разделе **Узлы (VPN nodes)** создайте узел:
   - `host` / `sni` — домен приложения (`<app>-<user>.<регион>.amvera.tech`
     или кастомный домен);
   - `port` — `443`;
   - `publicKey`/`shortId` — можно оставить пустыми (не используются для VLESS+WS).
4. Выданные ключи автоматически добавляются в конфиг Xray. Пользователям
   рекомендуется использовать **ссылку-подписку** (кнопка на странице ключей) —
   она автообновляется при смене конфигурации.
5. Чтобы пригласить первого пользователя — скопируйте свою реферальную ссылку
   из профиля и передайте ему. Без реферального кода регистрация заблокирована.

## Персистентность выданных ключей

`amvera.yml` монтирует `/etc/xray` как `persistenceMount` — список живых клиентов
Xray переживает передеплой: `entrypoint.sh` перерисовывает конфиг из шаблона,
но переносит список `clients` из предыдущего конфига на persistent-томе.

## Локальная сборка (проверка)

```sh
docker build -t vibe-proxy-nexus .

docker run --rm -p 8080:8080 \
  -e DATABASE_URL=... \
  -e SESSION_SECRET=... \
  -e ADMIN_EMAIL=admin@example.com \
  -e ADMIN_PASSWORD=changeme \
  vibe-proxy-nexus
```

## Отличие от `deploy/amvera-vpn-node`

- **`deploy/amvera-all-in-one`** (этот пакет) — всё в одном контейнере,
  VPN-транспорт VLESS+WebSocket на общем веб-домене. Используется в проде сейчас.
- **`deploy/amvera-vpn-node`** — только VPN-узел (Xray + management API),
  задел на будущее для масштабирования на несколько стран/регионов.
  Пока не подключён к основному бэкенду.
