# Vibe Proxy Nexus — единый контейнер для Amvera

Этот пакет собирает **весь проект в одном Docker-образе** для Amvera Cloud:

- **Xray-core** — VPN-сервер (VLESS), слушает только `127.0.0.1:10000` внутри
  контейнера, наружу напрямую не торчит;
- **Node/Express** — API, собранный веб-интерфейс **и** прокси
  WebSocket-апгрейда до Xray, всё на одном публичном порту (`8080`).

Оба процесса запускаются параллельно через `supervisord`. Поскольку бэкенд и
Xray находятся в одном контейнере, при выдаче/отзыве ключа бэкенд напрямую правит
конфиг Xray на диске и перезапускает процесс — отдельный management API не нужен.

Снаружи контейнера остаётся только:

- **PostgreSQL** — управляемая база Amvera (передаётся через `DATABASE_URL`).

Авторизация (email + пароль) реализована прямо в бэкенде и хранит сессии в
той же PostgreSQL — никаких внешних сервисов для входа не нужно.

Файлы `Dockerfile` и `amvera.yml` лежат в **корне репозитория** (Amvera по
умолчанию ищет их там); контекст сборки — корень репозитория.

## Почему VLESS едет поверх WebSocket, а не сырым TCP/Reality

Amvera-е edge (Traefik) на публичном HTTP(S)-порту **всегда** терминирует TLS
сам и ожидает за собой обычный HTTP. Это значит:

- **Reality несовместим в принципе** — протокол требует, чтобы именно Xray
  сам держал TLS-хендшейк (маскировка под реальный сайт), а тут TLS уже снят
  чужим прокси до того, как трафик доходит до Xray.
- **Сырой VLESS через TCP-контроллер Amvera** (домены `MONGO`/`POSTGRES`/
  `REDIS`, порты 27017/5432/6379) технически даёт настоящий TCP-passthrough
  без терминации TLS, и в одной из более ранних версий этого проекта
  использовался именно он. От этой схемы отказались в пользу варианта ниже —
  она требует отдельного TCP-домена, привязывает клиента к нестандартному
  порту и не так надёжна при проверках/апдейтах платформы.

Рабочая и используемая сейчас схема — **VLESS поверх WebSocket на обычном
HTTPS-домене**:

1. Xray слушает `127.0.0.1:10000` с настройками `network: ws`,
   `security: none`, `wsSettings.path: /vpnws` — это чистый локальный VLESS,
   без TLS (её берёт на себя Amvera снаружи).
2. Node-сервер (`artifacts/api-server/src/index.ts`) сам обрабатывает HTTP
   `upgrade`-события: если путь `/vpnws` и это настоящий WebSocket-хендшейк —
   открывает TCP-соединение к `127.0.0.1:10000` и прозрачно проксирует байты
   в обе стороны. Всё остальное — сразу `socket.destroy()`.
3. Клиент подключается как к обычному защищённому веб-сокету:
   `security=tls&type=ws&sni=<веб-домен приложения>&path=/vpnws` на порт
   `443`/`8080` (тот же домен и порт, что и сама панель). Amvera видит
   легитимный HTTPS/WebSocket-запрос и просто пробрасывает его в контейнер.

Итог: **не нужен отдельный TCP-домен, не нужен Dedicated IPv4, VPN и веб-панель
живут на одном обычном домене и порту**. Подробная история попыток (что не
сработало и почему) — `.agents/memory/amvera-raw-tcp-port.md`.

```
                 ┌───────────────────────── контейнер ─────────────────────────┐
  Браузер ──────▶│  Node/Express  :8080  ──▶  /api/*             (API)         │
  VPN-клиент ───▶│  (тот же порт)         ├─▶  /vpnws (WS upgrade) ─┐          │
                 │                        └─▶  всё остальное (React SPA)      │
                 │                                                  ▼          │
                 │                                    Xray-core :10000 (loopback)
                 │        ▲                                                    │
                 │        └── правит /etc/xray/config.json + supervisorctl     │
                 └──────────────────────────────────────────────────────────────┘
                          │
                          ▼
                  PostgreSQL (Amvera)
```

## Переменные окружения

### Секреты (в панели Amvera)

| Переменная       | Назначение                                                        |
| ---------------- | ------------------------------------------------------------------ |
| `DATABASE_URL`   | строка подключения к PostgreSQL от Amvera                          |
| `SESSION_SECRET` | секрет для подписи cookie сессии **и** для HMAC-подписи ссылок-подписок (`openssl rand -hex 32`) |
| `ADMIN_EMAIL`    | email первого администратора — создаётся автоматически при первом старте, если пользователя с таким email ещё нет |
| `ADMIN_PASSWORD` | пароль для этого администратора |

### Обычные значения

| Переменная | По умолчанию | Назначение |
| ---------- | ------------ | ---------- |
| `PORT`     | `8080`       | единственный публичный порт: веб-интерфейс + API + VPN (WebSocket-прокси до Xray) |

Ключи Reality/Xray (`publicKey`, `shortId`, `sni`, `host`, `port`) больше не
задаются переменными окружения контейнера — они хранятся в таблице
`vpn_nodes` и вводятся через админ-панель при создании узла (см. «После
деплоя» ниже).

## Подготовка перед первым деплоем

1. **База данных.** Создайте в Amvera управляемый PostgreSQL и возьмите его
   `DATABASE_URL`. Схему накатывать вручную не обязательно — `entrypoint.sh`
   делает `drizzle-kit push` в фоне при каждом старте контейнера
   (идемпотентно, безопасно при повторных запусках).

2. **Секрет сессии.** Сгенерируйте случайную строку для `SESSION_SECRET`:

   ```sh
   openssl rand -hex 32
   ```

3. **Учётные данные первого администратора** — задайте `ADMIN_EMAIL` и
   `ADMIN_PASSWORD` как секреты; аккаунт создастся сам при первом запуске.

## Деплой

Из Replit — командой `./deploy.sh "Сообщение на русском о том, что изменилось"`
(пушит в GitHub через API, т.к. `git push` в шелле агента заблокирован).
Дальше:

1. Amvera сама подхватывает пуш в GitHub и пересобирает образ по корневому
   `Dockerfile`.
2. Секреты (`DATABASE_URL`, `SESSION_SECRET`, `ADMIN_EMAIL`, `ADMIN_PASSWORD`)
   проставляются один раз в панели Amvera, раздел «Переменные» — `env:` в
   `amvera.yml` не поддерживается.

## После деплоя

1. Откройте веб-интерфейс по домену приложения Amvera, войдите под
   `ADMIN_EMAIL`/`ADMIN_PASSWORD`.
2. В разделе **Узлы (VPN nodes)** создайте узел:
   - `host` / `sni` — домен самого приложения (`<app>-<user>.<регион>.amvera.tech`
     или кастомный домен, если подключён);
   - `port` — `443` (обычный HTTPS-порт этого же домена);
   - `publicKey`/`shortId` — можно оставить пустыми (не используются в
     схеме VLESS+WS, это поля для Reality, если он когда-нибудь понадобится).
3. Выданные из панели ключи автоматически добавляются в живой конфиг Xray;
   пользователи могут либо скопировать отдельный `vless://`-линк, либо
   (рекомендуется) использовать самообновляющуюся **ссылку-подписку**
   (`/api/vpn-keys/subscription-url` в API, кнопка «Ссылка подписки» на
   странице ключей) — она подтягивает актуальный список ключей автоматически.

## Персистентность выданных ключей

`amvera.yml` монтирует `/etc/xray` как `persistenceMount`, поэтому список
живых клиентов Xray переживает передеплой: `entrypoint.sh` при каждом старте
перерисовывает конфиг из шаблона, но переносит в него список `clients` из
предыдущего конфига на persistent-томе.

## Локальная сборка (проверка)

```sh
docker build -t vibe-proxy-nexus .

docker run --rm -p 8080:8080 \
  -e DATABASE_URL=... \
  -e SESSION_SECRET=... \
  -e ADMIN_EMAIL=admin@example.com \
  -e ADMIN_PASSWORD=changeme \
  vibe-proxy-nexus
```

## Отличие от `deploy/amvera-vpn-node`

- **`deploy/amvera-all-in-one`** (этот пакет) — всё в одном контейнере,
  VPN-транспорт VLESS+WebSocket на общем веб-домене. Используется в проде
  сейчас.
- **`deploy/amvera-vpn-node`** — только VPN-узел (Xray + management API),
  задел на будущее для масштабирования на несколько стран/регионов, когда
  узлы выносятся на отдельные машины, а панель остаётся одна. Пока не
  подключён к основному бэкенду.
